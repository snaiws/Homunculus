### 핵심 설계 원칙
- **자기 수정 전체 허용**: TS 플러그인, 스킬, 지침, Python 모두 수정 가능
- **git 기반 안전장치**: 모든 수정은 브랜치에서 진행. commit 후 수정. 실패 시 `git revert`
- **멀티 인스턴스**: 각 호문쿨루스는 고유 instance ID를 가지고 자기 브랜치에서 작업
- **PR 기반 진화**: 호문쿨루스가 개선안을 PR로 제출. 사용자(또는 다른 인스턴스)가 리뷰/머지

## Phase별 목적 및 요구사항 정의

### Phase 1: 생존 (Survival)

**목적**: 호문쿨루스가 플라스크(서버) 안에서 **살아있는 상태**를 만든다.
OpenClaw을 기반으로 기본적으로 돌아가고, 사용자와 소통할 수 있으며,
후속 Phase를 위한 기초 설계(인스턴스 체계, 지침 프레임워크)를 올바르게 잡는다.

> 비유: 호문쿨루스가 플라스크에서 처음 눈을 뜨고, 주인을 인식하고, 대화할 수 있는 상태.

| # | 요구사항 | 상세 | 완료 기준 |
|---|---------|------|----------|
| P1-1 | OpenClaw 통합 | submodule로 OpenClaw 가져오기 | 빌드 성공, Gateway 시작 |
| P1-2 | Docker 배포 | docker-compose로 원커맨드 실행 | `docker-compose up`으로 전체 구동 |
| P1-3 | LLM 프로바이더 설정 | env 파일로 API 키 설정, 프로바이더 선택 | LLM 응답 가능 |
| P1-4 | 기본 채널 연결 | Discord 봇으로 명령 수신/응답 | Discord에서 대화 가능 |
| P1-5 | 부팅 시 환경 점검 | 네트워크/API/채널 자동 검증 | BOOT_REPORT.md 생성, 실패 시 콘솔 에러 |
| P1-6 | 채널 자동 감지 | 설정된 채널 중 연결 가능한 것 활성화 | 미연결 채널 자동 비활성화 |
| P1-7 | 기본 Workspace | AGENTS.md, SOUL.md 등 운영 파일 | 에이전트가 페르소나 인식 |
| P1-8 | 플러그인 스캐폴드 | @homunculus/core 빈 플러그인 | 플러그인 로드 로그 확인 |
| P1-9 | 인스턴스 식별 체계 | instance.yaml 설계 | 파일 존재 + 구조 정의 완료 |
| P1-10 | 지침 프레임워크 | 지침 작성 형식 정의 (배경/근거) | 형식 문서화 + 예시 지침 1개 |

### Phase 2: 자율 (Autonomy)

**목적**: 호문쿨루스가 **스스로 생활 리듬**을 갖는다.
스케줄에 따라 자고 일어나며, 자기 리소스를 관리하고,
스킬을 추가/관리할 수 있으며, 자기 코드를 수정하고 PR할 수 있다.

> 비유: 호문쿨루스가 하루의 루틴을 갖고, 필요한 도구를 스스로 만들어 쓰며,
> 자기 개선안을 주인에게 제안하는 단계.

| # | 요구사항 | 상세 | 완료 기준 |
|---|---------|------|----------|
| P2-1 | State Machine | Open/Close/Sleep 상태 전이 관리 | 상태 전이 + 디스크 영속 |
| P2-2 | 스케줄 엔진 | 크론잡 기반 수면/기상 사이클 | 설정 시간에 자동 수면/기상 |
| P2-3 | Directive 시스템 | YAML 지침 파일 로딩/검증/핫리로드 | 지침 변경 시 재시작 없이 반영 |
| P2-4 | 토큰 예산 관리 | 일일/야간 토큰 추적 & 게이팅 | 예산 초과 시 요청 거부 |
| P2-5 | 스킬 관리 메타 시스템 | 스킬 추가/삭제/목록 관리 스킬 | 대화로 새 스킬 생성 가능 |
| P2-6 | 자기 수정 시스템 | git branch + commit + rollback | 코드 수정 후 자동 커밋 + 실패 시 revert |
| P2-7 | PR 제출 시스템 | gh pr create로 메인 repo에 PR | 개선안 PR 생성 확인 |
| P2-8 | KakaoTalk 채널 | 카카오톡 플러그인 연동 | 카카오톡으로 대화 가능 |

### Phase 3: 성장 (Growth)

**목적**: 호문쿨루스가 **지능적으로 성장**한다.
Python으로 작업을 수행하고, 메모리를 능동적으로 정리하며,
사용자를 깊이 인식하고, 리소스를 효율적으로 관리한다.

> 비유: 호문쿨루스가 도구를 다루고, 경험에서 배우며,
> 밤에 하루를 정리하고 내일을 준비하는 단계.

| # | 요구사항 | 상세 | 완료 기준 |
|---|---------|------|----------|
| P3-1 | Python Executor | Python 코드 실행 도구 | Discord에서 Python 실행 결과 확인 |
| P3-2 | 리소스 모니터링 | 디스크/메모리/프로세스 감시 | 임계값 초과 시 경고 |
| P3-3 | 리소스 정리 (수면) | 야간 캐시/로그 정리 루틴 | 수면 시 자동 정리 실행 |
| P3-4 | 야간 메모리 통합 | 로그 → MEMORY.md 사실 추출 | 다일 운영 후 통합 결과 확인 |
| P3-5 | 사용자 프로필 | USER.md 구조화, 선호도 학습 | 사용자 패턴 인식 확인 |
| P3-6 | 롱텀 메모리 강화 | 시맨틱 검색 + 자동 큐레이션 | 과거 대화 내용 기억 확인 |

### Phase 4: 번식 (Propagation) — 향후

**목적**: 호문쿨루스가 **여러 인스턴스로 퍼져** 각자 진화하고 지식을 공유한다.

> 비유: 플라스크 속 호문쿨루스가 다른 플라스크에도 심어져,
> 각자 경험하고 발견한 것을 서로 PR로 공유하는 생태계.

| # | 요구사항 | 상세 |
|---|---------|------|
| P4-1 | 멀티 인스턴스 관리 | instance ID별 브랜치 네이밍, 충돌 방지 |
| P4-2 | 인스턴스 간 PR 리뷰 | 다른 인스턴스의 PR을 읽고 코멘트 |
| P4-3 | 지식 동기화 | MEMORY.md 간 유용한 사실 공유 |
| P4-4 | 적응형 스케줄링 | 사용자 활동 패턴 학습 → 자동 스케줄 조정 |
| P4-5 | 배포 자동화 | 새 인스턴스 원클릭 배포 (OCI/E2B/EC2) |

## 자기 수정 & 분산 진화 시스템 (R10, R11)

### Git 기반 자기 수정 플로우

```
1. 호문쿨루스가 개선 아이디어 발견
2. git checkout -b homunculus/{instance-id}/improve-{feature}
3. 코드/스킬/지침 수정
4. git commit -m "자동 개선: {설명}"
5. 자체 테스트 (수정된 코드 검증)
6. 실패 시: git revert HEAD → 원래 브랜치로 복귀
7. 성공 시: git push origin → gh pr create
```

### 멀티 인스턴스 구조

```
GitHub Repo (main)
  ├── homunculus/seoul-01/improve-memory     ← 서울 인스턴스 PR
  ├── homunculus/tokyo-02/add-skill-weather  ← 도쿄 인스턴스 PR
  └── homunculus/london-03/fix-sleep-cycle   ← 런던 인스턴스 PR
```

각 인스턴스는:
- `config/instance.yaml`에 고유 ID, 위치, 이름 설정
- 자기 브랜치 네임스페이스: `homunculus/{instance-id}/*`
- 다른 인스턴스의 PR을 읽고 리뷰 코멘트 가능 (Phase 4+)

### 롤백 매커니즘
- **즉시 롤백**: `git revert HEAD` (마지막 커밋 되돌리기)
- **특정 시점**: `git log` → `git revert {commit-hash}`
- **전체 복구**: `git reset --hard main` (메인 브랜치로 리셋)
- **자동 롤백 트리거**: 수정 후 헬스체크 실패 시 자동 revert
